# Podcast RSS Support Design

**Goal:** Add podcast RSS support by parsing playable media attachments and rendering them in article detail without enabling summary, translation, or full-text automation for podcast feeds.

**Context:**

FeedFuse already parses RSS and Atom feeds through `src/server/integrations/rss/parseFeed.ts`, stores entries as `articles`, and renders article content through the reader UI. Existing parsing extracts article images from `media:thumbnail`, image `media:content`, image `enclosure`, and `itunes:image`, but audio and video enclosures are currently ignored. The `articles` table has no media attachment model, and feed refresh currently enqueues filtering plus optional full-text, summary, and translation workflows for new RSS articles.

External references point to a stable podcast pattern: RSS 2.0 and Apple Podcasts represent episode media through `enclosure` attributes such as `url`, `length`, and `type`; Atom uses `link rel="enclosure"` with `href`, `type`, and `length`; Miniflux models enclosures independently and selects player-capable attachments by MIME prefix `audio/*` or `video/*`.

**Recommended Approach:**

Use a dedicated article media attachment model. Parse playable RSS/Atom enclosures into structured attachments, persist them separately from article text, expose them through article DTOs, and render the first playable attachment in `ArticleView`. Podcast feeds should be classified at fetch time when the parsed feed contains playable attachments, and every item from that refresh should skip full-text, AI summary, translation, and article filtering automation.

## Scope

In scope:

- Parse RSS `<enclosure>` elements and Atom `link rel="enclosure"` elements.
- Accept playable media only when the normalized URL is HTTP(S) and MIME type starts with `audio/` or `video/`.
- Persist article media attachments with URL, MIME type, optional size, optional duration, and stable ordering.
- Return media attachments from article detail APIs and API client DTOs.
- Render the first playable attachment in the article detail view with native browser controls.
- Prevent podcast RSS sources from entering full-text, summary, translation, and article filtering workflows.

Out of scope:

- Playback progress sync.
- Chapters, transcripts, Podcasting 2.0 advanced metadata, and episode funding metadata.
- Download management or offline playback.
- Separate podcast library/navigation UI.
- AI summary, body translation, title translation, full-text fetching, and article filtering for podcast RSS sources.

## Architecture

### RSS Integration

`parseFeed` should extend `ParsedFeedItem` with `mediaAttachments`. Each attachment should include:

- `url`
- `mimeType`
- `sizeBytes`
- `durationSeconds`

RSS parsing should read `enclosure` from `rss-parser` and also preserve Atom `link rel="enclosure"` through the existing `links` custom field. `itunes:duration` should be parsed into seconds when present. Invalid duration values should return `null`.

Image extraction remains separate. Image enclosures continue to feed `previewImage`; playable audio/video enclosures feed `mediaAttachments`.

### Persistence

Add `article_media_attachments`:

- `id bigint generated by default as identity primary key`
- `article_id bigint not null references articles(id) on delete cascade`
- `url text not null`
- `mime_type text not null`
- `size_bytes bigint null`
- `duration_seconds int null`
- `position int not null default 0`
- `created_at timestamptz not null default now()`

Indexes:

- `article_media_attachments_article_id_idx` on `article_id`
- unique `(article_id, url)` to avoid duplicate attachments for one article

Repository code should keep attachment writes near article writes. A small helper such as `insertArticleMediaAttachments` can batch insert attachments after `insertArticleIgnoreDuplicate` returns a newly created article.

### Feed Classification and Automation Skip

Podcast behavior should be derived from parsed feed content, not from a manual user flag in the first iteration. During feed refresh:

- If any parsed item has at least one playable attachment, treat the fetched source as a podcast source for this refresh.
- Store attachments for each created article that has playable attachments.
- For all created articles in that podcast refresh, do not enqueue full-text, AI summary, body translation, title translation, or article filtering jobs.
- Podcast episodes should still be inserted, listed, marked read/starred, and opened like normal articles.

This keeps normal RSS behavior unchanged while preventing media episodes from being sent into text-centric workflows.

### API and Frontend

Extend `ArticleDto` and shared `Article` types with:

```ts
mediaAttachments: Array<{
  id: string;
  url: string;
  mimeType: string;
  sizeBytes: number | null;
  durationSeconds: number | null;
}>;
```

Article detail API should include attachments. Reader snapshot/list payloads can omit attachments unless an existing selected-article flow requires them before detail load.

`ArticleView` should render a compact media player above article content when `mediaAttachments[0]` exists:

- `audio/*` -> `<audio controls preload="metadata">`
- `video/*` -> `<video controls preload="metadata">`

The player should use the direct enclosure URL. It should not go through the image proxy.

## Error Handling

- Drop attachments with empty URL, non-HTTP(S) URL, missing MIME type, or unsupported MIME type.
- Drop invalid `length`; save valid non-negative integers as `sizeBytes`.
- Drop invalid `itunes:duration`; save valid values as `durationSeconds`.
- If attachment insert fails after article creation, let the refresh fail normally so the feed fetch result records an error and can retry.
- Keep ordinary RSS feeds unaffected when there are no playable attachments.

## Open Decisions

- Do not add a manual `feed.kind = 'podcast'` or user-facing podcast source type in this iteration.
- Do not add playback progress storage yet.
- Do not proxy audio/video URLs unless a later security or CORS requirement demands it.
